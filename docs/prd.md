# CLAIM — MVP PRD

> **설계사가 제안하고, AI 가 비교하고, 당신은 선택합니다.**
>
> CLAIM 은 가입자가 한 번의 요청을 남기면 검증된 여러 설계사로부터 제안서를 받고, AI 가 약관 기준으로 객관 비교한 결과를 한 자리에서 확인할 수 있게 하는 보험 매칭 서비스. 추천에 어떠한 금전적 이해관계도 없는 중립 매칭을 보장한다.

- 작성일: 2026-05-05 (결과 화면 디자인 확정: 2026-05-17)
- 상태: MVP 범위 확정, 결과 화면 구현 완료 (시나리오 chip + ROI log 차트 + 보장 패널 + 질병 검색 모달)
- 관련 문서: [architecture.md](./architecture.md), [pages.md](./pages.md)

## 0. 브랜드 & 서비스 정체

- **브랜드명**: CLAIM
- **한 줄 슬로건**: 설계사가 제안하고, AI 가 비교하고, 당신은 선택합니다.
- **핵심 가치**:
  - **한 번의 요청** — 가입자는 짧은 요청서 한 장으로 흩어진 비교 과정을 끝낸다.
  - **여러 제안서** — 검증된 설계사가 가입자에게 직접 제안서를 보낸다.
  - **객관 비교** — AI 가 약관 기준으로 보장 / 회수 배율 / 보장 밸런스를 비교해 보여준다.
  - **중립 추천** — 광고비·수수료에 영향받지 않는 형평성 기반 매칭.
- **3-act 사용자 경험**:
  1. 설계사가 **제안한다** — 알림톡으로 받은 요청에 정형 필드 + PDF + 메모 제출.
  2. AI 가 **비교한다** — 모든 제안서를 약관 기준으로 정형화하고 다축으로 시각화.
  3. 사용자가 **선택한다** — 비교 결과를 보고 마음에 드는 설계사에게 직접 연락.

## 1. 배경 & 문제

보험 가입자가 자신에게 맞는 설계사를 찾고 제안서를 받기까지의 과정은 분산되어 있고, 비교가 어렵다. 가입자는 여러 채널을 떠돌며 비슷한 정보를 반복 입력하고, 설계사는 매칭 기회의 편차가 크다.

CLAIM 은 가입자가 **한 번의 요청서**로 **여러 설계사의 제안서를 동시에 비교**할 수 있게 한다. 추천에 어떠한 금전적 이해관계도 없는 중립 매칭을 보장한다.

## 2. 목표 & 비목표

### 목표 (MVP)
- 가입자가 30초 안에 매칭 요청서를 제출할 수 있다.
- 가입자는 추천된 후보 중 복수의 설계사를 선택해 제안서를 동시에 요청할 수 있다.
- 설계사는 알림톡 한 번으로 즉시 작성 페이지에 진입해 제안서를 제출한다.
- AI가 도착한 제안서를 분석/취합하고, 가입자는 알림톡 링크로 결과를 확인한다.
- 모든 설계사가 매칭 기회를 고루 가진다.

### 비목표 (out of scope)
- 가입자 인앱 채팅, 회원 계정, 마이페이지 (향후 앱 출시 시점)
- 설계사 ↔ 가입자 후속 대화의 서비스 내 추적 (외부 채널에 위임)
- 제안 화면의 추천 강도/UI 디테일 (별도 논의)
- 알림톡 링크 보안 정책 디테일 (별도 논의, 기본 토큰만 적용)
- 보험 가입 자체 (우리는 매칭/제안서 전달까지)

## 3. 사용자

| 역할 | 정의 | 인증 |
|---|---|---|
| **가입자(Customer)** | 보험을 알아보는 일반 사용자 | 계정 없음. 휴대폰 번호가 식별자. 동의 시점에 SMS OTP 1회. |
| **설계사(Partner)** | 운영자가 사전 검증·등록한 보험 설계사 | 계정 + 비밀번호. 단, 알림톡 진입 시 일회용 토큰 URL로 로그인 우회 가능. |
| **운영자(Admin)** | 서비스 운영자 | 단일 어드민 계정. |

## 4. 핵심 사용자 흐름

### 4.1 가입자 흐름

```
[1] 요청서 작성 (비로그인)
       ↓
[2] N명 후보 노출 → 가입자가 K명까지 복수 선택
       ↓
[3] 정보 제공 동의 + 제안서용 정보 입력 + SMS OTP 인증
       ↓
[4] 선택된 설계사들에게 요청 송부 (알림톡)
       ↓
[5] 제안서 도착 (전원 제출 OR 제한시간 T 만료)
       ↓
[6] AI 분석/취합 → 가입자에게 알림톡 발송
       ↓
[7] 가입자가 알림톡 URL 진입 → 결과 열람
       ↓
[8] 마음에 드는 설계사에게 문자 (추후 디자인)
```

### 4.2 설계사 흐름

```
[1] 운영자가 풀에 등록 (사전 검증 완료)
       ↓
[2] 매칭 시 알림톡 수신 (일회용 토큰 URL 포함)
       ↓
[3] URL 진입 → 작성 페이지 직진 (로그인 우회)
       ↓
[4] 정형 필드 + PDF + 자유 메모 제출
       ↓
[5] 제한시간 T 내 미제출 시 자동 미제출 처리 (별도 거절 UI 없음)
```

### 4.3 운영자 흐름

- 설계사 풀 CRUD (등록/수정/비활성화)
- 시스템 설정값 변경 (N, K, T, 페널티 윈도우)
- 요청 모니터링 (상태 추적, 0명 제출 케이스 알림)

## 5. 기능 명세

### 5.1 요청서 작성 (1단계)

가입자가 비로그인 상태에서 작성. 입력 시간 ~30초 목표.

| 필드 | 형태 | 필수 |
|---|---|---|
| 관심 보장 카테고리 | 다중 선택 (칩) | ✓ |
| 가입 목적 | 선택지 + 직접입력 | ✓ |
| 연령대 | 10대 단위 | ✓ |
| 성별 | M/F | ✓ |
| 거주 지역 | 시/도 | ✓ |
| 월 예상 보험료 범위 | 슬라이더 | ✓ |
| 휴대폰 번호 | 숫자 입력 (검증은 3단계 OTP에서) | ✓ |

**제약**: 같은 휴대폰 번호로 진행 중인 요청이 있으면 **신규 요청 차단** + 안내 메시지.

### 5.2 매칭 & 후보 선택 (2단계)

#### 매칭 알고리즘

**필터**: 설계사의 전문 보험 2개 중 요청 카테고리와 1개 이상 일치

**정렬 (우선순위 순)**:
1. **누적 노출 횟수 적은 순** — 형평성 핵심. 신규 설계사가 자연스럽게 상위.
2. **미제출률 낮은 순** — 동률 깨기. 페널티 윈도우 K건 기준 (`미제출 수 / K`). 이력 없는 신규는 0점으로 유리.
3. **랜덤 셔플** — 최종 동점 처리.

상위 **N명**(서버 설정값) 노출.

#### 가입자에게 노출되는 설계사 정보 (카드)
- 사진
- 이름
- 전문 보험 2개
- 한줄 소개

#### 선택
가입자는 후보 중 **최대 K명**(서버 설정값)까지 복수 선택 가능.

### 5.3 정보 제공 동의 + 최종 생성 (3단계)

가입자가 선택을 확정하면 다음 단계 진입:

#### 제안서용 추가 정보 입력

| 필드 | 형태 | 필수 |
|---|---|---|
| 생년월일 | 날짜 | ✓ |
| 직업 | 텍스트 | ✓ |
| 흡연 여부 | Y/N | ✓ |
| 키 / 몸무게 | 숫자 | ✓ |
| 기존 보험 가입 여부 | Y/N + 자유 메모 | ✓ |
| 최근 병력 | 간단 체크 + 자유 메모 | 선택 |

#### 동의
- 개인정보 제3자 제공 동의 (선택한 설계사들에게 위 정보 + 1단계 정보 전달)
- 통신 수신 동의 (알림톡, SMS)

#### SMS OTP 인증
6자리 코드. 인증 완료 시 요청서 최종 생성 + 설계사들에게 송부.

### 5.4 설계사 제안서 제출

#### 알림 채널
새 요청 도착 시 **카카오 알림톡** 발송. 본문에 일회용 토큰 URL 포함.

#### 진입 방식
- **알림톡 URL** → 토큰 검증 후 작성 페이지 직진 (로그인 불필요)
- **일반 로그인** → 받은 요청 목록 → 해당 요청 작성 페이지

토큰은 1회 사용 + 제한시간 T 내 만료.

#### 제출 폼 (정형 필드 + PDF + 메모)

**카테고리 무관 공통 필드:**
- 월 보험료
- 총 납입 기간
- 총 보장 금액 (대표 1개)
- 핵심 담보 3슬롯 (자유 입력)
- 갱신/비갱신
- 만기/환급 여부

**+ PDF 첨부 (실제 제안서)**
**+ 자유 메모**

#### 미제출 처리
제한시간 T 내 미제출 시 자동 "미제출" 상태. 별도 거절 UI 없음. 미제출 이력은 페널티 윈도우에 누적.

### 5.5 트리거 & AI 분석

#### 트리거 조건 (incremental)
> **제안서가 1건 도착할 때마다 즉시 그 제안서 분석 시작.**
>
> - 첫 제출 시: plan_request.status `dispatched` → `analyzing`
> - 추가 제출: 그대로 `analyzing` 유지하며 새 제안서도 분석 큐에 투입
> - 모든 assignment 가 종결 (submitted 또는 expired) + 제출된 모든 제안서 분석 완료 시: `completed` 로 전환 + 가입자에게 결과 알림톡

장점: 마지막 설계사를 기다리느라 가입자가 결과를 늦게 받는 일 없음.

#### AI 분석/취합
- 입력: 가입자 요청서 + 도착한 모든 제안서 (정형 필드 + PDF 텍스트 + 자유 메모)
- 출력: 비교/취합 결과 + 추천. 결과 화면 디자인은 별도 논의.

#### 알림톡 발송 (가입자 대상)
- 본문 형태: **"제안서 N건이 도착했습니다" + URL** (옵션 1, 미리보기 정보 미포함)
- URL은 결과 열람 페이지로 deeplink
- 발송 시점: 모든 분석이 끝나 `completed` 전환 시점

### 5.6 결과 열람 + 후속 액션

확정 + 구현된 흐름:
- 가입자가 알림톡 URL로 진입해 모든 제안서를 한 자리에서 확인.
- **설계사 chip 탭** 으로 제안서 전환. 본문 통째 교체.
- **시나리오 chip** = admin 우선순위 × 모든 제안서 교집합 top-3 + 검색(`+`) chip.
  - 검색 chip → 질병 선택 모달 (한글 자소 분리 검색 지원).
  - 모달 선택 시 가장 오래된 chip 이 빠지고 새 카테고리가 가장 오른쪽 추가 (FIFO).
- **ROI(회수 배율) 차트** = log10 y축. `보장액 / 누적 보험료(납기 반영)` 으로 시나리오별 시계열.
- **해지 시 월부담 차트** = 분석 리포트의 refund_table 시계열을 (납입 − 환급) ÷ 가입 후 경과 개월로 환산한 월평균. 가입~만기 구간으로 clip.
- **보장 패널** = 활성 시나리오의 모든 담보 항목 (담보명 말줄임 + 가격 nowrap, `total_insured_amount` = ROI 분자와 정확히 일치).
- 마음에 드는 설계사에게 즉시 문자 발송 (이후 대화는 외부).

여전히 보류:
- 알림톡 URL 접근 통제 강도 (현재는 token 검증만)
- 만료된 결과 토큰의 DB row 삭제 자동화 (현재는 lazy 검사 — `dispatchedAt + 결과 보관 기간` 경과 시 만료 화면 노출, row 는 보존)

### 5.7 엣지 케이스 — 비어있을 때

#### 0명 제출 (전원 미제출)
1. **1회 자동 재매칭** — 새 N명 후보 추천 → 가입자에게 알림톡으로 재선택 요청. 요청서/동의 정보 재사용 (가입자 추가 입력 없음).
2. 재매칭도 0명 제출이면 → 사과 알림톡 + "지금은 매칭이 어려웠습니다" 안내. 운영자 모니터링에 별도 표시.

#### 일부만 제출 (K명 중 X명, X ≥ 1)
- 도착한 X건만 그대로 결과 화면에 노출.
- 알림톡 본문은 "제안서 X건이 도착했습니다"로 정확히 표기.

### 5.8 어드민

단일 운영자 계정. 최소 기능만:

#### 설계사 풀 관리
| 필드 | 가입자 노출 | 매칭 사용 |
|---|---|---|
| 사진 | ✓ | — |
| 이름 | ✓ | — |
| 전문 보험 2개 | ✓ | ✓ (필터) |
| 한줄 소개 | ✓ | — |
| 휴대폰 번호 (알림톡) | — | — |
| 로그인 계정 (이메일/비번) | — | — |
| 활성화 여부 | — | ✓ (비활성 = 매칭 제외) |

CRUD + 비활성화 기능. 검증은 운영자가 오프라인에서 수행 후 등록.

#### 시스템 설정값

DB `app_settings` 테이블 (single row, id='app') 에 저장. 어드민 시스템 설정 화면에서 mutate. 호출부는 [`server/settings.ts`](../src/server/settings.ts) 의 `getSettings()` 로 매 요청 로드 (값 인라인 X — 어드민 변경 직후 반영).

| 컬럼 | 의미 | 기본값 |
|---|---|---|
| `candidateCount` (N) | 가입자에게 노출할 후보 수 | 5 |
| `selectLimit` (K) | 가입자가 선택 가능한 최대 인원 | 3 |
| `submissionDeadlineHours` (T) | 설계사 제안서 제출 제한시간 (시간) | 48 |
| `penaltyWindow` | 미제출률 페널티 산정 윈도우 (최근 N건) | 10 |
| `resultRetentionDays` | 결과 페이지 토큰 보관 기간 (일). `dispatchedAt + N일` 경과 시 결과 화면 만료 (lazy 검사, DB row 는 보존) | 7 |
| `scenarioPriority` | 결과 페이지 시나리오 chip 의 우선순위 (`category_payouts.category` ordered list) — 모든 제안서 교집합과 비교해 보장되는 상위 3 가 chip 초기값 | `[]` |

#### 요청 모니터링
- 진행 중 요청 목록 (상태: 요청됨 / 매칭됨 / 제출중 / AI분석중 / 완료 / 재매칭 / 종료)
- 0명 제출되어 자동 재매칭 트리거된 케이스 별도 표시 → 수동 개입 여지

## 7. 비기능 요구

### 7.1 보안 / 개인정보
- 가입자 제안서용 정보는 **선택된 설계사들에게만** 전달. 후보 노출 시점에는 1단계 정보만.
- 알림톡 토큰: 1회 사용 + T 시간 내 만료. (강도는 추후 보강)
- PDF 파일: 가입자 요청과 무관한 제3자 접근 불가.
- 가입자 휴대폰 번호는 마스킹된 형태로 어드민에 표시.

### 7.2 성능
- 1단계 → 2단계 매칭 결과 노출까지 < 1초 (P95)
- AI 분석 트리거 → 알림톡 발송까지 < 60초 (P95)

### 7.3 모니터링
- 어드민 모니터링 화면에서 운영자가 진행 상태 추적
- 자동 재매칭 트리거 / 0명 제출 종료 케이스는 알림 강조

## 9. 성공 지표

### 북극성 지표
**연결 전환율** = (마음에 드는 설계사에게 문자를 보낸 가입자 수) / (제안서를 1건 이상 받은 가입자 수)

→ 이 시점이 우리가 제공하는 진짜 가치가 발생하는 순간.

### 보조 지표
| 지표 | 정의 |
|---|---|
| 요청서 완료율 | 1단계 시작 / 제출 완료 |
| 매칭 동의율 | 후보를 본 가입자 / 동의+OTP 통과 |
| 설계사 제출률 | 받은 요청 / 제한시간 내 제출 (= 1 − 미제출률) |
| 제안서 도착률 | 가입자 기준, 1건 이상 받은 비율 |

## 11. 용어집

| 용어 | 의미 |
|---|---|
| **제안서** | 설계사가 작성한 실제 가입 가능한 보험 설계안. 단순 견적과 구분. |
| **요청서** | 가입자가 작성하는 매칭 요청. 1단계(매칭용) + 3단계(제안서용)로 분리됨. |
| **매칭** | 시스템이 풀에서 N명 후보를 노출하고, 가입자가 K명을 선택하는 과정. |
| **풀(Pool)** | 운영자가 사전 검증·등록한 설계사 집단. 매칭 후보 풀. |
| **알림톡** | 카카오톡 비즈니스 메시지. 가입자/설계사에게 시스템 알림 채널로 사용. |
| **N / K / T** | 각각 후보 수 / 선택 상한 / 제출 제한시간. 모두 서버 설정값. |
